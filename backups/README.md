# HRMS Backups Directory

## Overview
This directory contains database backup files generated by the `db-backup.sh` script (Phase 9 implementation). Backups are essential for disaster recovery, data migration, and historical analysis.

## 📁 Directory Structure

```
backups/
├── README.md                  # This documentation
├── hrms_backup_20260410_103000.sql    # Example backup file
├── hrms_backup_20260409_220000.sql    # Previous backup
├── hrms_backup_20260408_220000.sql    # Older backup
└── .gitkeep                  # Keep directory in git (empty files ignored)
```

## 🔄 Backup Strategy

### Backup Schedule
- **Daily Backups**: Automatic nightly backups (recommended)
- **Weekly Full Backups**: Complete database dumps
- **Monthly Archives**: Long-term retention for compliance
- **On-Demand Backups**: Before major changes or deployments

### Retention Policy
- **Daily Backups**: Keep for 7 days
- **Weekly Backups**: Keep for 4 weeks
- **Monthly Backups**: Keep for 12 months
- **Yearly Backups**: Keep for 7 years (compliance)

### Backup Naming Convention
```
hrms_backup_YYYYMMDD_HHMMSS.sql
```
- **YYYY**: Year (4 digits)
- **MM**: Month (2 digits)
- **DD**: Day (2 digits)
- **HH**: Hour (24-hour format)
- **MM**: Minute (2 digits)
- **SS**: Second (2 digits)

## 🚀 Using Backups

### Creating a Backup
```bash
# Using the backup script (Phase 9)
./scripts/db-backup.sh backup

# Manual backup
docker-compose exec -T postgres pg_dump -U hrms_user hrms > backups/hrms_backup_$(date +%Y%m%d_%H%M%S).sql
```

### Restoring from Backup
```bash
# Using the restore script (Phase 9)
./scripts/db-backup.sh restore backups/hrms_backup_20260410_103000.sql

# Manual restore
docker-compose exec -T postgres psql -U hrms_user -d hrms < backups/hrms_backup_20260410_103000.sql
```

### Listing Available Backups
```bash
# Using the list command (Phase 9)
./scripts/db-backup.sh list

# Manual listing
ls -la backups/*.sql
```

## 📊 Backup Contents

### What's Included
- **Database Schema**: All tables, indexes, constraints
- **Data**: All rows from all tables
- **Sequences**: Current values of serial columns
- **Permissions**: User roles and grants (if using --clean)
- **Extensions**: PostgreSQL extensions (if any)

### What's Not Included
- **PostgreSQL configuration files**
- **File system data** (BLOBs stored outside database)
- **External dependencies**
- **Application code**

## 🔒 Security Considerations

### Backup Security
1. **Encryption**: Backup files contain sensitive data - encrypt in production
2. **Access Control**: Restrict read/write access to authorized personnel
3. **Transfer Security**: Use secure protocols (SFTP, SCP) for remote backups
4. **Storage Security**: Store backups in secure, access-controlled locations

### Production Best Practices
- **Offsite Storage**: Keep copies in geographically separate locations
- **Testing**: Regularly test backup restoration procedures
- **Monitoring**: Monitor backup job success/failure
- **Documentation**: Maintain detailed recovery procedures

## 🚨 Disaster Recovery

### Recovery Scenarios

#### 1. Data Corruption
```bash
# Identify corruption
psql -U postgres -d hrms_db -c "VACUUM VERBOSE ANALYZE;"

# Restore from backup
./scripts/db-backup.sh restore backups/hrms_backup_20260410_103000.sql
```

#### 2. Accidental Deletion
```bash
# Stop application
docker-compose down

# Restore database
./scripts/db-backup.sh restore backups/hrms_backup_20260410_103000.sql

# Start application
docker-compose up -d
```

#### 3. Hardware Failure
```bash
# On new server, restore from backup
scp user@old-server:/path/to/hrms/backups/hrms_backup_latest.sql .
./scripts/db-backup.sh restore hrms_backup_latest.sql
```

### Recovery Time Objectives (RTO)
- **Critical Data**: 1 hour maximum downtime
- **Important Data**: 4 hours maximum downtime
- **Historical Data**: 24 hours maximum downtime

### Recovery Point Objectives (RPO)
- **Critical Data**: 15 minutes maximum data loss
- **Important Data**: 1 hour maximum data loss
- **Historical Data**: 24 hours maximum data loss

## 🔧 Advanced Backup Options

### Incremental Backups
```bash
# Using WAL archiving (PostgreSQL)
# Configure in postgresql.conf:
wal_level = replica
archive_mode = on
archive_command = 'cp %p /path/to/wal_archive/%f'
```

### Point-in-Time Recovery (PITR)
```bash
# Base backup
pg_basebackup -D /path/to/base_backup -U postgres

# Combined with WAL archiving for PITR
# Can recover to specific timestamp
```

### Cloud Backups
```bash
# AWS S3 backup
aws s3 cp backups/hrms_backup_latest.sql s3://hrms-backups/

# Azure Blob Storage
az storage blob upload --file backups/hrms_backup_latest.sql --container-name backups

# Google Cloud Storage
gsutil cp backups/hrms_backup_latest.sql gs://hrms-backups/
```

## 🐳 Docker-Specific Considerations

### Volume Backups
```bash
# Backup PostgreSQL data volume
docker run --rm -v hrms_postgres_data:/data -v $(pwd)/backups:/backup alpine tar czf /backup/postgres_data_$(date +%Y%m%d).tar.gz -C /data .

# Restore volume from backup
docker run --rm -v hrms_postgres_data:/data -v $(pwd)/backups:/backup alpine sh -c "rm -rf /data/* && tar xzf /backup/postgres_data_latest.tar.gz -C /data"
```

### Docker Compose Backups
```bash
# Backup all services
docker-compose exec -T postgres pg_dump -U hrms_user hrms > backup.sql
docker cp hrms-backend:/app/logs/ ./backend_logs_backup/
docker cp hrms-frontend:/usr/share/nginx/html/ ./frontend_backup/
```

## 📞 Support & Troubleshooting

### Common Issues

#### Backup Failed
```bash
# Check PostgreSQL service
docker-compose ps postgres

# Check disk space
df -h .

# Check permissions
ls -la backups/

# Check logs
docker-compose logs postgres
```

#### Restore Failed
```bash
# Check backup file integrity
head -n 5 backups/hrms_backup_latest.sql

# Check database connection
docker-compose exec postgres psql -U hrms_user -d hrms -c "SELECT 1"

# Check for existing data conflicts
docker-compose exec postgres psql -U hrms_user -d hrms -c "\dt"
```

#### Backup File Too Large
```bash
# Compress backup
gzip backups/hrms_backup_latest.sql

# Exclude large tables (if appropriate)
docker-compose exec -T postgres pg_dump -U hrms_user hrms --exclude-table-data=audit_logs > backup.sql

# Split backup
split -b 100M backups/hrms_backup_latest.sql backups/hrms_backup_latest.sql.part
```
